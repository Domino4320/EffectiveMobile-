# Бэкэнд-приложение - система аутентификации и авторизации. Приложение реализовано в ходе выполнения тестового задания от компании EffectiveMobile на позицмю Junior Python разработчика
____
## О проекте
____
### Описание
Приложение представляет собой API, которое включает в себя механизмы аутентификации и авторизации. В данном прложении пользователь может зарегестрироваться, а также изменять свой профиль и удалить его (если он аутентифицирован и авторизован в системе). Также приложение предусматривает роль админа, который может ограничивать действия конкретных пользователей и групп пользователей.
____
### Использованные технологии
В качестве основного фреймворка я выбирал Django REST Framework (DRF) и СУБД PostgreSQL. В PostgreSQL я уже раннее работал в совем предыдущем пет-проекте, однако DRF был для меня "новизной" и я работал с ним впервые. Я использовал пакеты drf-spectacular и drf-spectacular-sidecar для подключения интерактивной документации (Swagger и Redoc). В качестве инструмента для контроля зависимостей я выбрал poetry, т.к. с ним очень удобно работать с пакетами и установкой их зависимостей. Для удобного подключения к базе данных и работы с .env файлами был использован pydantic settings.
____
### Как устроена база данных и система управления доступом в моем приложении
Так как в ТЗ было указано не использовать решения "из коробки" пришлось продумывать свою структуру БД. 
В первую очередь нужно было создать таблицу пользователей, где будут храниться их пароли в захэшированном виде. Структура таблицы пользователей:
- id
- имя
- фамилия
- отчество
- email
- пароль
- статус (показатель того, удален пользователь или нет с помощью soft delete)
- роль
____
О ролях я скажу чуть позже в этом же разделе, т.к. пришел к этому я не сразу. Далее по ТЗ было сказано, что я должен реализовать возможно редактирования данных пользователя, также удаление через механизм soft delete (для которого и был введен статус). Чтобы реализовать вышеописанные функции, прежде всего, нужно было ввести механизм аутентификации для приложения. В качестве механизма я выбрал следующую схему - токены и cookie (httponly для повышения безопасности).
#### Почему я выбрал именно данный вариант?
Для начала стоит отметить какие варианты еще присутствовали:
- Сессии
- Bearer аутентификация с JWT токенами

Сессии показались мне решение "из коробки", а также у них доавольно нагруженный способ передачи данных серверу, поэтому я их не выбрал, но от части мой кастомный механизм с токенами напоминает работу сессий. JWT токены же показались мне сложными по части logout, ведь для того, чтобы их отозвать нужно было бы создавать черные списки для jwt токенов и обращаться каждый раз за дополнительной информацией о них в БД (что по идеи jwt токены не предполагают). Да конечно в связке с jwt токенами для logout можно было бы использовать redis и тогда проблем было бы гораздо меньше, но я решил воспользовать более сбалансированным вариантом.
#### Как работает мой механизм аутентификации
Для начала я описал модель токена через DjangoORM. Вот его структура:
- id
- строка токена
- дата истечения
- какому пользователю принадлежит
При каждом логине сервер формирует объект токена и заносит его в базу данных, но браузер тоже должен знать о нашем токене - с этим мне помогли cookie (почему браузер, а не клиент - потому что я хотел чтобы аутентификация была защищена от воздействия программным способом через JavaScript код). В функции логина я заносил в cookie через встроенные механизмы DRF токен, его дату истечения. Таким образом мне удалось аутентифицировать пользователей в системе.
#### Кастомный middleware для аутентификации
Для того, чтобы не прописывать в каждой вьюшке получение данных о пользователе через cookie, я сделал свой middleware, который получает данные из cookie, отправляет запрос в базу данных и получает пользователя, в противном случае - выдает ошибку 401 (Not Authorized). В этом middleware я также прописал некоторые исключения, чтобы он не проверял url, которые не требуют аутентификации. В случае успешной аутентификации пользователя мой middleware добавляет к объекту request аттрибут user, к которому могут обращаться въюшки.
#### Небольшое заключение по аутентификации
После всей проделанной работы я создал две вьюшки, для изменения данных о пользователе (PATCH запрос) и для удаления пользователя (DELETE запрос).
#### Роли и права доступа
Далее по ТЗ требовалось создать API для администраторов, через которое можно выдавать права доступа. Для этого нуэно было описать таблицу ролей для лучшего масштабирования. Структура таблицы:
- id
- имя роли
Помимо этого требовалось создать еще 3 таблицы - таблица с ресурсами приложения (на которые и будут накладывать ограничения администраторы), таблица с правами доступа для конкретных пользователей и таблица с правами доступа для ролей (да в моем приложении есть только пользователи и администраторы, но в целях удобства масштабирования я сделал для этого отдельную таблицу).
Структура таблицы с ресурсами:
- id
- название ресурса
_____
Структура таблицы с правами доступа для конкретных пользователей:
- id
- уровень доступа
- id пользователя
- id ресурса
____
Структура таблицы с правами доступа для ролей:
- id
- уровень доступа
- id роли
- id ресурса
_____
Уровень доступа - это число, которое представляет собой, что может делать пользователь на данном ресурсе:
- 0 - никаких прав
- 1 - только чтение
- 2 - запись, чтение, обновление
- 3 - полный crud
- 4 - администраторские права
#### Кастомный middleware для авторизации
С вышесозданными таблицами я уже мог полноценно продумывать систему авторизации проекта. Для этого я снова выбрал создать кастомный middleware. В нем я снова определил пути исключения, которые ему трогать не нужно. Данный midlleware просматривал тип запроса, пользователя и ресурс на который запрос направляется. С помощью этих данных он либо передавал управление вьюшке, либо выкидывал ошибку 403 (Forbidden) в случае если у пользователя недостаточно прав.

### Как запустить проект
1. Скачайте архив, либо воспользуйтесь командой git clone для установки проекта
2. Создайте базу данных в PostgreSQL
3. Пропишите в .env файле в папке с auth_service (именно в той, в которой лежит settings.py) данные для подключения к базе данных (DB_USER, DB_HOST, DB_PORT, DB_PASSWORD, DB_NAME - именно с такими же названиями, чтобы не конфликтовать с pydantic settings)
4. Установите зависимости через poetry install
5. Примените миграции через python3 manage.py migrate (находясь в папке auth_service)
6. Запустите сервер через python3 manage.py runserver (находясь в папке auth_service)
